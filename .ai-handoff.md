## Key Decisions Made
- Use Spotify Client Credentials for public track search — it avoids exposing the client secret and does not require login for catalog search.
- Use Spotify OAuth Authorization Code flow for real playback control — playback and queue APIs require a user access token.
- Use httpOnly cookies for Spotify user tokens — safer than exposing access/refresh tokens to the frontend.
- Use `http://127.0.0.1:3000/api/auth/callback` for local redirect URI — Spotify's current redirect URI rules reject `localhost`; loopback IP literals may use HTTP.
- Keep the Spotify iframe embed as visual/fallback playback UI — the real Premium playback happens through the Web API against the user's active Spotify device.
- Queue behavior now uses Spotify's real queue endpoint — previous `PUT /me/player/play` with a URI array did not reliably create a visible Spotify queue. The route now starts the selected track, then calls `POST /me/player/queue` for the rest.
- Visible search results are the source of the generated queue — if results are `A, B, C, D` and user clicks `B`, playback starts with `B`, then queues `C, D, A`.
